# 바이브 코딩 실전 사례 연구
## 남양주 스마트주정차 앱 — 처음부터 끝까지 전체 기록

> **대상 독자**: 코딩을 전혀 모르거나 초보자인데 AI와 함께 실제 프로덕션 앱을 만들고 싶은 분  
> **핵심 메시지**: 개발 용어를 몰라도, 하고 싶은 것을 말로 설명하면 됩니다.

---

## 바이브 코딩이란?

코드를 직접 작성하는 대신, **원하는 것을 말로 설명**하고 AI가 구현하는 협업 방식입니다.

```
사람이 하는 일           AI가 하는 일
─────────────────        ─────────────────────────────
"이게 안 돼"             원인 분석 → 코드 수정
"이런 기능 넣어줘"        설계 → 구현 → 파일 저장
"배포해줘"               버전 업 → 빌드 → GitHub 업로드
"이게 맞아?"             코드 검증 → 문제 있으면 재수정
```

핵심은 **실제로 실행해보고 피드백을 주는 것**입니다.

## 대화 1 — 방향 설정: "상용 프로그램처럼 만들고 싶어"

### 사용자 첫 마디

> "남양주 아키텍처 개선안은 너의 이전 버전이 작성한 거야. 내가 원하는 건 Electron 버전으로 기존 HTML 파일을 전환해서 단일 HTML 파일을 넘어, 로그인/통계/대시보드/지도 탭 등 실제 상용화 프로그램처럼 만들어두고 DATA를 따로 DATA 폴더에 두어서 관리하고 싶어."

### 핵심 요구사항 정리

- 기존: 지도만 있는 단순 HTML 파일
- 목표: 로그인 → 메인화면 → 탭 전환 구조의 완성형 앱
- 데이터는 `data/` 폴더에서 별도 관리

### AI가 제안한 단계별 계획 (Phase 구조)

```
Phase 0: Electron 프로젝트 구조 세팅
Phase 1: 앱 기반 안정화 (IPC, 보안, 파일 구조)
Phase 2: 로그인 + 탭 네비게이션 구조
Phase 3: 데이터 관리 (Excel 파일 읽기/가져오기)
Phase 4: DB 연동 (CCTV, 단속 데이터 ↔ 대시보드/통계)
Phase 5: 사이드바 UI/UX 전면 개선
Phase 6: 설정 탭 추가
```

### 사용자의 승인 방식

> "좋아 페이즈0부터 단계별로 실행하자. 가장 중요한 점 1. 앱 안정성 2. 코드 오류 없이 3. 다양한 변수를 감안하여."

**바이브 코딩 포인트**: 큰 그림을 먼저 합의하고, 단계별로 진행했습니다. 처음부터 모든 걸 한꺼번에 만들려 하지 않았습니다.

## 대화 2~3 — Phase 0, 1 실행

### Phase 0: Electron 프로젝트 세팅

AI가 한 일:
- `package.json` 생성 (Electron, better-sqlite3, electron-builder 등 의존성 정의)
- `main.js` 메인 프로세스 파일 생성
- `preload.js` 보안 브릿지 설정 (contextIsolation: true)
- 폴더 구조 생성: `src/`, `renderer/`, `static/`, `data/`

### Phase 1: 앱 기반 안정화

- IPC(프로세스 간 통신) 채널 설계
- `src/ipc/handlers.js` — 렌더러↔메인 데이터 통신 설정
- `src/db/database.js` — SQLite 데이터베이스 초기화

**바이브 코딩 포인트**: IPC, contextIsolation 같은 기술 용어를 사용자는 몰라도 됩니다. "앱 안정성 우선"이라는 방향만 제시했고, AI가 보안 설계를 알아서 했습니다.

## 대화 4~6 — Phase 2: "앱을 열면 로그인이 나와야지"

### 사용자 피드백

> "아키텍처 페이즈2에 큰 다른점이 있어. 우리가 보통 앱을 열면 로그인 창이 나오고 → 로그인 후 메인화면 → 메인화면에서 탭을 선택해서 대시보드, 통계창, 지도탭 이렇게 들어가야 하는데, 지금은 그냥 대시보드 탭이랑 통계탭만 추가된 수준이고 심지어 데이터 연동도 안 되어서."

### 사용자가 지적한 것

1. 로그인 화면이 없음
2. 탭 전환 구조가 없음
3. 데이터 연동 안 됨

### AI가 구현한 것

- 로그인 오버레이 (아이디: admin, 비밀번호: 1234)
- 로그인 → 메인화면 전환 애니메이션
- 탭 네비게이션: 지도 / 대시보드 / 통계 / 리포트 / 데이터관리 / 설정
- `static/js/modules/auth.js` — 인증 모듈 분리

**바이브 코딩 포인트**: "보통 앱을 열면 로그인이 나오잖아" 라는 일상적인 표현으로 완성형 UI 구조를 요청했습니다.

## 대화 7~11 — Phase 3, 4: 데이터 관리 + DB 연동

### Phase 3: Excel 파일 읽기

사용자 요청 전 현명한 질문:

> "페이즈3 진행하기 전에 데이터 양이 많잖아, 특히 엑셀 파일 같은 경우에. 이 경우에 실제로 문제 해결을 위한 솔루션을 개발자들은 어떻게 하는지 알아?"

→ AI가 실제 개발 현장의 방법론(청크 처리, 가상 스크롤, 백그라운드 import 등)을 설명  
→ 사용자가 전략을 이해한 뒤 구현 승인

**구현 결과**: xlsx 라이브러리로 Excel 파일 읽기, DB에 저장, 진행률 표시

### Phase 4 후 발견된 문제

> "데이터관리탭에서 엑셀 파일 3개 가져오는 데까지는 성공했어. 문제점 1. CCTV 데이터는 목록에 없음 2. 대시보드 및 통계탭에 데이터관리탭에서 업로드된 엑셀 파일에 연동하여 변화가 있어야 하는데 아무런 변화가 없음. 코드 수정 전에 원인 분석 및 계획안 먼저 짜봐."

**바이브 코딩 포인트**: "코드 수정 전에 원인 분석 먼저" — 바로 고치는 게 아니라 **이해하고 계획한 뒤 수정**하는 패턴을 반복했습니다. 이것이 버그 재발을 막는 핵심 습관입니다.

## 대화 12~16 — Phase 5: 사이드바 전면 리디자인

### 사용자의 요청 방식

이 시점에서 사용자는 매우 구체적인 역할을 AI에게 부여했습니다:

> "너는 10년 차 Senior Frontend Developer이자 B2B SaaS(특히 GIS/지도 기반 서비스) 전문 UX/UI 디자이너야. 현재 개발 중인 '지도 탭'의 좌측 사이드바(Sidebar) UI/UX를 상용 솔루션 수준으로 전면 리팩토링할 거야."

### AI의 접근

전문가 역할을 부여받은 AI가 스스로 계획을 수립:
- 레이어 관리 섹션 (CCTV, 탄력운영구간, 어린이보호구역, 버스전용차로, 읍면동 경계)
- CCTV 목록 페이지네이션 (266개 전체 → 상위 10개 + 더보기)
- 통합검색 (도로명/지번/CCTV 통합)
- 단속 파일 업로드 기능

**바이브 코딩 포인트**: AI에게 **전문가 페르소나**를 부여하면 더 높은 수준의 결과물이 나옵니다. "10년 차 시니어처럼 해줘"가 일반적인 요청보다 훨씬 좋은 결과를 냅니다.

## 대화 22~25 — 성능 최적화 (Phase A, B)

### 사용자 피드백

> "이거 지도탭에서도 그렇고 로그인 탭에서도 그렇고 처음 로딩이 길어서 그런 건지 텍스트를 입력해야 하는 칸에 커서를 두고 눌러도 반응이 한참 동안이나 없어."

### AI의 접근 방식

1단계: 코드 수정 전 **심층 분석**부터

> 가능한 원인들:
> - Excel 파일을 메인 스레드에서 동기 처리 → UI 블로킹
> - V-World 지도 초기화가 완료될 때까지 대기
> - SQLite 쿼리 동기 실행

2단계: Phase A (빠른 수정들) 실행 → 테스트

> "아직도 좀 느리네. 정상적인 속도는 아니야. 10초는 넘어가니까 페이즈 B 진행해줘."

3단계: Phase B (심층 최적화) 실행

> "좋아 이제 좀 빨라졌네."

**바이브 코딩 포인트**: "빠르게 해줘" 한 마디지만, AI가 스스로 단계를 나누어 점진적으로 개선했습니다. 중간에 피드백("아직도 느리네")을 주는 것만으로 다음 단계가 진행됩니다.

## 대화 26~28 — 통합검색 개선

### 문제

> "지도 탭에 통합검색이 제대로 작동을 안 해. 도로명 주소, 지번 주소, CCTV 이름 검색을 통합하고 싶은데 검색이 마커만 검색해주는 것 같고 그마저도 오류가 나와."

2. 검색 결과 앞에 `(지번)`, `(도로명)` 구분 표시 추가
3. V-World 검색 API와 DB 검색 통합

## 대화 29~32 — 사이드바 UX 개선

> "지도 탭에서 고정형 전체가 266개가 너무 길어서 사이드바가 너무 길어져. 상위 10개만 나오고 나머지는 > 눌러서 다음 10개."

> "처음 시작할 때 로딩바가 나오고 완료 시 → 로그인화면 이렇게 구성할 수 있어?"

> "대시보드 첫 번째 줄에는 CCTV 관련(전체/운영중/미운영), 두 번째 줄에는 전체단속-고정형-주행형-시민신고."

**바이브 코딩 포인트**: 이 구간에서 요청이 매우 구체적이고 짧습니다. 사용자가 앱을 직접 써보면서 불편한 점을 즉각 피드백으로 주고 있습니다. 이것이 바이브 코딩의 핵심 리듬입니다.

## 대화 33~36 — 지도 초기화 + API 경고 제거

> "검색에서 청평면 경춘로 1037이 있고 남양주시 경춘로 1037이 있거든. 겹치는 곳은 로직이 어떻게 되는 거야? 한 번 알아만 봐줄래. 코딩 수정은 하지 말고."

→ 원인 파악: 시/군 경계 없이 검색하면 인접 시의 결과도 포함  
→ 해결: 검색어에 "남양주시" 포함하면 해결됨  
→ 코드 수정 없이 사용 방법으로 해결

> "앱 새로고침이 너무 무거워. 그냥 지도만 초기화하는 로직 해줄 수 없어?"

> "처음 시작할 때 API 안 맞는다고 경고창 뜨는데 이 경고창 자체를 안 뜨게 할 수 있어?"

---

## 대화 37~46 — 통계 탭: 날짜 필터 기능

### 요청

> "통계탭에 날짜를 지정할 수 있는 칸을 넣어서 그 날짜마다 통계를 보여주는 거지. 처음 시작할 때는 최근 12개월로 고정."

### 문제 발생 및 해결 과정

구현 후 테스트:

> "날짜를 변경하고 조회해도 그대로야. 내가 봤을 때는 날짜를 못 읽어서 그런 것 같아, 엑셀 내에서 말이야. 기획안부터 작성해봐, 코딩 수정하지 말고."

→ AI가 날짜 포맷 분석 → 실제 Excel의 날짜 형식 확인 → 파싱 로직 수정

빠른 날짜 선택 버튼 추가:

### 패턴 1: "코드 수정 전에 계획 먼저"
```
사용자: "문제가 있어. 코드 수정 전에 원인 분석 및 계획안 먼저 짜봐."
→ 이해 없이 수정하면 버그가 재발합니다.
→ 분석 → 계획 → 승인 → 구현 순서를 지키세요.
```

### 패턴 2: 오류 메시지는 그대로 복사
```
사용자: "SqliteError: unable to open database file at C:\Program Files\..."
→ 기술 용어를 해석하려 하지 마세요.
→ 오류 메시지 전체를 그대로 붙여넣으면 AI가 해석합니다.
```

### 패턴 3: 실제로 써보고 피드백
```
사용자: "-1년 버튼 누르면 24년 12월 30일부터 25년 12월 31일, 이런 식이야."
→ "틀렸어"가 아니라 실제로 어떻게 됐는지 설명합니다.
→ 스크린샷보다 동작 설명이 더 효과적입니다.
```

### 패턴 4: 방향을 물어보는 것도 전략
```
사용자: "앱 안정성 기준으로 일반적으로 추가하는 기능이 뭐가 있어?"
→ 모를 때 물어보면 AI가 옵션을 제시합니다.
→ 사람은 그 중에서 선택하고 우선순위를 정합니다.
```

### 패턴 5: 전문가 페르소나 부여
```
사용자: "너는 10년 차 Senior Frontend Developer이자 GIS 전문 UX/UI 디자이너야."
→ 역할을 구체적으로 부여할수록 결과물의 수준이 높아집니다.
```

### 패턴 6: 틀린 것은 즉시 지적
```
사용자: "지도 기반이 국토교통부 V-World인데? 카카오 맵이라고 잘못 되어 있어."
→ AI가 틀렸을 때 바로 지적하면 코드를 확인하고 수정합니다.
→ 틀린 채로 넘어가면 나중에 더 큰 문제가 됩니다.
```

### 패턴 7: 작게 배포하고 반복
```
v1.0.1 → v1.0.2 → v1.0.3 → ... → v1.0.8
→ 한 번에 완벽하게 만들려 하지 않습니다.
→ 조금씩 만들고, 실제로 써보고, 고치는 반복이 핵심입니다.
```
